#### 数据基建

系统要做的事只有四件——知道有哪些款式和它们的排名、知道用户试了什么、知道外部在流行什么、知道AI做了哪些决定。四件事对应四张核心表，加一张审计表。

|styles  |款式主表（C端展示、权重排序）|
|---|---|
|users  |用户表（\&\#34;我的\&\#34;模块基础）|
|try\_on\_logs|试戴记录（含user\_id）|
|favorites   |收藏夹|
|external\_trends|爬虫\+VLM打标结果|
|daily\_reports|LLM日报\+调权指令|
|operation\_logs|权重变更审计|

#### styles（款式主表）

- 系统的\&\#34;商品库\&\#34;，也是 C 端推荐排序的直接依据。

|字段|类型|说明|
|---|---|---|
|id|INTEGER PK|主键|
|name|TEXT|款式展示名称|
|color|String|颜色标签，如\&\#34;冰透粉\&\#34;|
|design|String|图案类型|
|shape|String|甲形，如\&\#34;方形\&\#34;\&\#34;椭圆\&\#34;|
|style|String|风格，如\&\#34;甜美少女\&\#34;|
|image\_url|TEXT|款式图URL|
|rank\_weight|REAL|推荐权重，默认1\.0，越高越靠前|
|is\_active|BOOLEAN|是否上架展示|
|created\_at|TIMESTAMP|创建时间|

#### try\_on\_logs（试戴行为日志）

- 只记录\&\#34;谁在什么时候试了哪款\&\#34;。

|字段|类型|说明|
|---|---|---|
|id|INTEGER PK|主键|
|style\_id|INTEGER FK→styles|试戴的款式|
|tried\_at|TIMESTAMP|试戴时间戳|
|user\_id|INTEGER FK→users, nullable|允许匿名试戴（未登录用户）|

#### external\_trends（外部趋势，已打标）

爬虫抓回来后立即经过 VLM 打标，把结构化结果存进来。

|字段|类型|说明|
|---|---|---|
|id|INTEGER PK|主键|
|platform|TEXT|试戴的款式|
|tried\_at|TIMESTAMP|\&\#34;xiaohongshu\&\#34; / \&\#34;douyin\&\#34;|
|post\_url|TEXT UNIQUE|帖子原始链接（去重用，防止重复爬）|
|engagement|INTEGER|点赞\+收藏数（热度信号）|
|color|TEXT|VLM识别的颜色标签|
|design|TEXT|VLM识别的图案标签|
|shape|TEXT|VLM识别的甲形标签|
|style|TEXT|VLM识别的风格标签|
|scraped\_at|TIMESTAMP|抓取时间|
|posted\_at|TIMESTAMP|帖子在平台的原始发布时间|

#### daily\_reports（日报与调权指令）

- LLM 每天生成一次，输出两样东西：给人看的日报文本，和给机器执行的 JSON 指令。两者存同一行。

|字段|类型|说明|
|---|---|---|
|id|INTEGER PK|主键|
|report\_date|DATE UNIQUE|日期（每天唯一一条）|
|content\_md|TEXT|日报正文（Markdown格式）|
|strategy\_json|TEXT|JSON字符串|
|status|TEXT|pending/approved/rejected/executed|
|created\_at|TIMESTAMP|生成时间|

#### operation\_logs（操作审计日志）

- 每次实际修改 `rank\_weight` 后写一条记录。用于 B 端展示操作历史和支持手动回滚。

|字段|类型|说明|
|---|---|---|
|id|INTEGER PK|主键|
|report\_id|INTEGER FK→daily\_reports||
|style\_id|INTEGER FK→styles|被操作的款式|
|weight\_before|REAL|修改前权重|
|weight\_after|REAL|修改后权重|
|source|TEXT|\&\#34;auto\&\#34;（定时任务）/ \&\#34;manual\&\#34;（人工）|
|executed\_at|TIMESTAMP|执行时间|

#### users（用户表）

|字段|类型|说明|
|---|---|---|
|id|INTEGER PK|主键|
|nickname|TEXT|昵称|
|avatar\_url|TEXT|头像地址|
|is\_member|BOOLEAN|是否会员，默认False|
|member\_expires\_at|TIMESTAMP|会员到期时间，非会员为NULL（非必要）<br>|
|created\_at|TIMESTAMP|注册时间（非必要）|

#### favorites（收藏夹）

|字段|类型|说明|
|---|---|---|
|id|INTEGER PK|主键|
|user\_id|NTEGER FK→users|谁收藏的|
|style\_id|INTEGER FK→styles|收藏哪款|
|saved\_at|TIMESTAMP|收藏时间|

```Python
实现函数 compute_all_weights(db) -> list[dict]
对全部 is_active=True 的款式，每款计算新的 rank_weight。

【步骤1】准备数据：
  - 查每款近7天试戴量 → 用于热度排名
  - 查每款近3天、前3天试戴量 → 算growth_rate（公式同summary）
  - 查external_trends近14天数据，按平台和posted_at分出
    emerging(<=3天)和established(4-14天)两组

【步骤2】对每款计算三项得分：

内部热度分（满分2.5）：先把所有款式按近7天试戴量降序排名，再映射：
  排名1→2.5  排名2→2.0  排名3→1.5  排名4-5→1.0
  排名6-15→0.5  排名16+→0.2
  特殊：近7天试戴=0但历史试过→0.0；从未试戴(从未出现在try_on_logs)→0.1

内部动能分（满分1.5）：按growth_rate映射：
  >=200→1.5  100~200→1.0  50~100→0.6  0~50→0.3  <0→0.0

外部共振分（满分1.0）：检查该款(design,color)与外部数据的匹配，取最高适用项：
  design和color都命中任一emerging项→1.0
  仅design命中任一emerging项→0.7
  design和color都命中任一established项→0.6
  仅design命中任一established项→0.3
  无任何design匹配→0.0

【步骤3】计算最终权重：
  new_weight = 热度分 + 动能分 + 共振分
  new_weight = min(max(new_weight, 0.5), 5.0)

返回列表，每项：
{
  "style_id": int, "name": str,
  "old_weight": float,  // 当前styles表中的rank_weight
  "new_weight": float,  // 新计算值
  "heat_score": float, "momentum_score": float, "external_score": float,
  "change_reason": str  // 简述，如"热度第1+爆发增长+外部双榜共振"
}
注意：本函数只计算并返回，【不写入数据库】，写库由reports接口在人工审核后执行。

   - strategy_json.weight_changes包含全部在架款式，每条含三项得分明细
   - new_weight均在[0.5, 5.0]范围内
   - 试戴量第1的款式new_weight应接近5.0
   - content_md为纯文字日报，不含权重数字罗列
   - daily_reports新增一条status="pending"记录
```





其他：

后端接口和前端的串联逻辑：唯一需要你们提前约定的只有两件事：一是**接口地址和字段名**（比如返回的 JSON 里款式 id 叫 `id` 还是 `style\_id`），二是**本地联调时的端口**（你的 FastAPI 默认跑在 8000，前端开发服务器可能跑在 3000，需要你在 FastAPI 里开 CORS，这个之前的 Cursor prompt 里已经包含了）。
